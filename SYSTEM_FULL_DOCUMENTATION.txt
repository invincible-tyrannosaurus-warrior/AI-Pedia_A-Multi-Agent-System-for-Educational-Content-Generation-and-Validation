2. SYSTEM GOAL AND PRODUCT DEFINITION
========================================================================

2.1 System Goal
The system automatically builds educational content packages from a topic
prompt and optional uploaded files.

2.2 Generated Output Types
- Code demonstrations (Python scripts)
- Slides (PPTX)
- Narrated video lecture (MP4)
- Quiz package (JSON)

2.3 Design Philosophy
- Orchestration over monolith: each content type has a dedicated agent
- Deterministic where possible: PPT build is code-driven from storyboard
- Continuous feedback: streamed progress and step-level status in UI
- Verifiability: acceptance criteria and judging loop
- Reproducibility: run-scoped output folders and per-run logs

6. ORCHESTRATION CORE (task_manager_agent.py)
========================================================================

6.1 Core Responsibility
Central runtime brain that controls:
- planning
- task dependency scheduling
- pipeline invocation
- judging
- retries
- stream event emission
- run log persistence

6.2 Key Data Models

AcceptanceCriterion
- criterion_id
- criterion_type: mcp_tool | output_shape | file_exists | semantic
- target
- operator: equals | contains | exists | gte | lte
- expected
- severity: required | optional
- tool (for mcp_tool)
- evidence

SubTask
- task_id
- agent
- instruction
- inputs
- dependencies
- acceptance_criteria

TaskPlan
- user_intent
- subtasks

6.3 Registered Agents
- coder
- video
- presentation
- quizzer
- text is currently disabled in default registry use

6.4 Execution Modes

Mode 1: run_workflow(...)
- full plan generation through LLM tool-calling
- full judging rounds over all tasks
- may execute independent tasks in parallel via ThreadPoolExecutor
- useful for non-stream batch style operation

Mode 2: stream_workflow(...)
- deterministic task assembly based on UI toggles
- streams plan and runtime events for web client
- executes tasks in dependency order, with per-task retries
- applies judging after each task attempt

6.5 Why Two Modes Exist
- run_workflow is planning-centric and evaluator-heavy
- stream_workflow is UX-centric and deterministic for frontend
- both write run-level logs and support judging

6.6 Retry and Dependency Logic
- each task has retry budget
- failed tasks can receive fix instructions from judger
- downstream tasks depend on successful completion of declared dependencies
- blocked tasks are marked fail with dependency reason

6.7 Run Directory Convention
Each run writes into:
- data/generated_code/run_<run_id>/

Task inputs are normalized with:
- output_subdir = run_<run_id>
- output_filename for coder tasks

6.8 Refinement Logic
Function: refine_stream(run_id, task_id, feedback)

Flow:
1. Load prior run log from logs/task_manager/<run_id>.json
2. Restore assets and original task context
3. Append user feedback to instruction as critical update
4. Re-run only target agent
5. Emit updated artifact
6. Append refinement record into run log

========================================================================
7. JUDGER CORE (judger_pipeline.py)
========================================================================

7.1 Judger Purpose
Evaluates whether each task result satisfies acceptance criteria.

7.2 Criterion Processing

Deterministic criteria:
- file_exists
- output_shape
- mcp_tool

Semantic criteria:
- evaluated via LLM using structured payload

7.3 Deterministic Evaluation Engine
Important helper functions:
- _resolve_path
- _resolve_target
- _resolve_placeholders
- _apply_operator
- _evaluate_nonsemantic_criterion

Supported operators:
- exists
- equals
- contains
- gte
- lte

7.4 MCP Tool Integration
When criterion_type is mcp_tool:
- tool definition includes name + args
- placeholders in args are resolved from task result
- tool calls are cached per task to avoid repeated expensive calls

7.5 LLM Semantic Evaluation
Semantic checks use:
- model from JUDGER_MODEL (default gpt-5.2)
- strict tool schema "judge_task"
- returns verdict, failed criteria, evidence, fix instructions

7.6 Verdict Composition
- required failures from deterministic + semantic checks produce fail
- no required failures produce pass
- overall status:
  - pass: all tasks pass
  - partial: mixed pass/fail
  - fail: no passing tasks

10.1 Files

11. VIDEO PIPELINE
========================================================================

11.1 Files
- moe_layer/video_agent/video_pipeline.py
- moe_layer/video_agent/ppt_converter.py
- moe_layer/video_agent/script_writer.py
- moe_layer/video_agent/video_composer.py

11.2 End-to-End Flow
1. Resolve source PPTX from dependency artifacts or uploaded assets
2. Convert PPTX to PDF (libreoffice)
3. Convert PDF pages to PNG slides (PyMuPDF)
4. Generate narration script for each slide (vision LLM)
5. Generate speech audio per script (edge-tts)
6. Render per-slide MP4 segments (ffmpeg)
7. Concatenate segments into final lecture MP4

11.3 Stage-Based Progress Model
Stages:
- prepare
- convert_slides
- generate_scripts
- build_clips
- encode_video

Reporter computes weighted overall progress and emits payloads used by UI.

11.4 Runtime Requirements
- libreoffice installed and discoverable on PATH
- ffmpeg installed and discoverable on PATH
- edge-tts Python package installed
- PyMuPDF installed

11.5 Error Model
Any stage failure returns:
- success: false
- error: message
And pushes stage error update to progress reporter.

========================================================================
23. KNOWN LIMITATIONS AND RISKS
========================================================================

23.1 Dual Orchestration Paths
The system has both run_workflow and stream_workflow paths.
Behavior differs between them (planner-driven vs deterministic task build).

23.2 Text Agent Inactive
Text generator is currently a placeholder and not active in default registry.

23.3 Heavy Runtime Dependencies
Video path relies on libreoffice + ffmpeg + edge-tts + PyMuPDF.
Missing any of these blocks video generation.

23.4 External API Dependence
Most high-value stages depend on cloud LLM availability and valid API keys.

23.5 Legacy Generated Outputs
Historical generated scripts and artifacts may include non-portable paths
or dependencies; treat them as outputs, not authoritative source code.

========================================================================
25. CONCLUSION
========================================================================

AI Pedia Local Stream is a modular multi-agent educational generation
platform with strong orchestration visibility, run traceability, and
artifact-level refinement.

Its implementation combines:
- practical local file management
- cloud model orchestration
- deterministic content assembly where possible
- structured acceptance validation
- real-time user-facing progress telemetry

This makes it suitable for final year project demonstration as a complete
system rather than an isolated model demo.
